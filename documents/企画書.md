# LexFlow Protocol — 企画書

**AIが契約を読み、Ethereumが契約を実行し、JPYCが資金を動かす。**

---

## 1. 解決したいPain

### 🎯 ターゲット
企業法務（M&A / 資金調達領域の弁護士・VC・スタートアップ）

### ❗ 現場の構造的な不便（Pain）

| プロセス | Pain（不便） | 現状の問題点 |
|---------|-------------|-------------|
| **契約締結後の履行管理** | 条件の進捗確認・立証が属人的 | Slack/メール/電話での確認 → 手戻り/遅延 |
| **支払処理/成功報酬** | 決済が契約条件と同期せず | 遅延・未収が発生<br>「契約は終わったが支払いはまだ」が恒常化 |
| **AI活用の対価化** | AIレビュー/判定の成果と報酬の紐付け不可 | AI活用→利益構造に結びつきづらい |
| **国際/スタートアップ案件** | 送金/銀行/為替/営業時間の摩擦 | 決済が案件のボトルネックに |

**📌 契約・義務・決済が連動せず、弁護士業務全体に摩擦が生まれている。**

---

## 2. ソリューション概要

### 🎯 プロダクト：LexFlow Protocol
**契約条件 × AI判定 × JPYC決済の自動同期インフラ**

### 💡 仕組みの核心

```
PDF契約書 → AI解析（LangGraph）
           ↓ 条項/義務/支払条件を抽出
履行証跡/マイルストーンをAI判定
           ↓
Ethereumスマートコントラクト起動
           ↓
JPYC自動送金（エスクロー / 比例送金）
           ↓
取引証跡はオンチェーンに記録
```

### 🧪 使用技術（実装想定）
- **LangGraph**（AIエージェントフロー）
- **OpenAI API / Embeddings**（条項抽出）
- **Solidity**（エスクロー / 条件処理）
- **MetaMask / Hardhat**（MVP開発）
- **JPYC**（円建て安定決済）

**📌 ST（証券トークン）は今回範囲外 → フォーカス明確化**

## 3. 実現に向けた具体的アプローチ（Feasibility）

### 🛠️ 技術的ステップ

| 工程 | 内容 | 提出先で評価される観点 |
|-----|------|---------------------|
| **契約解析** | LangGraphで条項抽出/条件スキーマ化 | AI賞：AIエージェント活用が明確 |
| **履行判定** | 証跡入力→AIがYes/No（自動承認） | 実現可能性の高さ |
| **決済実行** | ハッシュ化IDをキーにSC起動/JPYC送金 | Ethereum賞：MVP稼働/Github提出を想定 |
| **証跡化** | Txログ/Etherscanで確認可能に | 透明性 × 法務実務適合 |

### 📈 ビジネス導入ステップ
1. 既存契約書そのまま（PDF）利用
2. 最終承認＝弁護士のUI検証で責任制御
3. 稼働案件から請求フローを自動化
